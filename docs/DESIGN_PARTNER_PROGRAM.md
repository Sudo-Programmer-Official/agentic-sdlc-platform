# Design Partner Program

## Purpose

This document defines how to run the early **Design Partner Program** for the Agentic SDLC Platform.

The purpose of the program is to:

- recruit the right early customers
- validate the strongest product wedge
- learn from real workflows
- gather proof points and product feedback
- convert pilots into recurring revenue

This is the most important near-term commercial motion.

## Program Thesis

The platform should not try to sell broad autonomy first.

It should first prove value in a narrow, credible workflow:

**bounded, reviewable AI-assisted software delivery**

That is the right design-partner promise because it aligns with what the product can genuinely support today.

## Ideal Design Partner Profile

The best early design partners are:

- startups with technical founders
- product teams with 3 to 15 engineers
- agencies shipping repeated client work
- engineering leaders exploring AI but frustrated by black-box workflows

The best partners already feel this pain:

- AI is useful but hard to trust
- current workflows are fragmented
- teams want more speed without repo chaos
- review and validation still matter

## Who to Avoid Early

Avoid partners who:

- expect fully autonomous end-to-end product creation
- need heavy enterprise security commitments immediately
- want large bespoke consulting engagements unrelated to the core wedge
- have no engineering ownership on their side

Those customers create learning noise and long cycles.

## Program Offer

### Duration

- 4 to 8 weeks

### Scope

- one primary workflow
- optionally one secondary workflow
- one repository or codebase at first

### Example pilot workflows

- bounded frontend feature implementation
- controlled backend feature work
- test generation for a scoped feature
- reviewable implementation run with validation and approval

### Deliverables

- workflow setup
- repo onboarding
- agreed run constraints
- pilot checkpoints
- end-of-pilot summary with findings and next recommendation

## Program Structure

### Phase 1: Qualification

Confirm:

- the team already builds software actively
- there is a clear owner on the customer side
- there is at least one bounded workflow worth piloting
- the partner is willing to give feedback and share outcomes

### Phase 2: Discovery and Workflow Definition

Define:

- target use case
- repo and environment scope
- success criteria
- what will not be attempted during the pilot

### Phase 3: Setup

Configure:

- repo onboarding
- workflow controls
- validation commands
- file scope expectations
- approval expectations

### Phase 4: Pilot Execution

Run:

- selected software tasks
- bounded implementation cycles
- review and validation loops
- weekly checkpoints

### Phase 5: Outcome Review

Assess:

- whether the workflow saved time
- whether trust improved
- whether review quality improved
- whether the team wants to continue

## Recommended Success Criteria

Every pilot should agree on a small set of measurable outcomes.

Recommended success metrics:

- number of successful reviewable runs
- time from request to reviewable output
- percentage of runs that passed bounded validation
- reduction in manual cleanup after AI-generated work
- number of workflows the team wants to repeat after the pilot

Qualitative signals also matter:

- did the team trust the system more than raw AI coding tools
- did the workflow feel more controllable
- did engineering leads feel comfortable with the review model

## Customer Commitments

To keep the program serious, each design partner should commit to:

- a technical owner
- regular feedback sessions
- access to at least one meaningful software workflow
- willingness to evaluate the product honestly

## Our Commitments

We should commit to:

- responsive onboarding
- close support during the pilot
- clear scoping and honesty about maturity
- product iteration based on partner feedback
- a concise end-of-pilot review

## What the Pilot Is Not

The pilot is not:

- a custom software agency engagement
- an unlimited consulting arrangement
- a promise of full autonomous product delivery
- a substitute for the partner's own engineering judgment

Keeping those boundaries clear protects both product focus and trust.

## Conversion Path After the Pilot

At the end of a successful pilot, the next step should be one of:

- Startup Team Plan
- Growth Engineering Plan
- extended design-partner agreement if the fit is strong but the product still needs iteration

## Suggested Pilot Closing Questions

Ask every design partner:

- Which workflow delivered the most value?
- Where did the system feel most trustworthy?
- Where did it still feel immature?
- What would need to improve for broader adoption?
- Would you pay to keep using this on the next set of changes?

## Why This Program Matters

The design-partner program is not only a sales tactic.

It is the fastest way to:

- validate the wedge
- improve the product with real feedback
- build credible case studies
- learn buying language
- prove that the platform solves a real operational problem
